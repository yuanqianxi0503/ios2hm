# Skill：iOS 任务总览笔记

## 我的职责

生成或修正本次迁移任务的 iOS 功能点总览笔记（.ios-notes/task-overview.md）。
支持两种模式：全量生成和按用户反馈补充修正。

---

## 执行入口

从调用方 prompt 中读取 mode 参数：

mode = initial    → 执行【Initial 模式】
mode = supplement → 执行【Supplement 模式】

---
---

## Initial 模式：全量生成任务总览

深度分析 iOS 项目中本次迁移任务涉及的全部代码，
生成一份功能点完整、无遗漏、无死代码的任务总览笔记。

### 输入

读取以下文件：
- .migration/request.md：任务描述、iOS 项目路径、用户提供的功能点列表（若有）

用户提供的功能点列表仅作参考线索，不直接采用。

---

### 执行步骤

**Step 1：定位任务入口，建立文件清单**

使用 Glob 和 Grep 在 iOS 项目中找到与任务相关的所有入口文件。

寻找策略（多管齐下，不只找主 ViewController）：
- 按任务描述中的关键词搜索文件名和类名
- 搜索 TabBar 配置、路由注册、页面跳转中与任务相关的注册点
- 若用户提供了功能点线索，以线索关键词为起点向外扩散搜索

建立初始文件清单：记录找到的所有入口文件路径。

---

**Step 2：全量 UI 交互扫描**

这是防止遗漏功能点的核心步骤。

对初始文件清单中的每个文件，逐一扫描所有可交互元素：

扫描对象（不能遗漏任何一类）：
- UIButton / SwiftUI Button 的 action / closure
- UIBarButtonItem（导航栏按钮，容易被遗漏）
- UITapGestureRecognizer / UILongPressGestureRecognizer 等手势
- UITableView / UICollectionView 的 didSelectRowAt
- 下拉刷新 UIRefreshControl 的 addTarget
- 上拉加载（scrollViewDidScroll 中的触底判断）
- 滑动手势（左滑删除、右滑操作等）
- 输入框的 delegate 方法（搜索框、筛选输入）
- segmentedControl / tabBar 的切换事件
- 页面生命周期中主动触发的逻辑（viewDidLoad 中的数据加载、viewWillAppear 中的刷新）
- 通知监听（NotificationCenter.addObserver）
- KVO 监听

对每个交互元素记录：
- 所在文件和行号
- 交互类型
- 绑定的处理方法名
- 处理方法是否有实际逻辑（下一步判断）

若扫描中发现新文件（如跳转目标页、被调用的子组件），
将新文件加入文件清单，对新文件同样执行本步骤。
循环直到文件清单不再增长。

---

**Step 3：死代码识别**

对 Step 2 扫描到的每个交互元素，判断其可达性：

**判断规则：**

标记为 DEAD（跳过，不作为功能点）：
- 处理方法存在但方法体为空（`{ }`）或只有注释
- 处理方法存在但在整个项目中零调用（用 Grep 全局搜索方法名确认）
- 被硬编码条件永久屏蔽：
  `if false`、`guard false`、`if (0)`
  `if FeatureFlag.xxx`，且 xxx 在项目中硬编码为 false/0/关闭
  `#if false` 预编译条件

标记为 UNCERTAIN（保留为功能点，在笔记中注明疑虑）：
- 处理方法体只有 TODO / FIXME 注释
- 被 Feature Flag 控制，但 Flag 值来自远程配置或运行时，无法静态判断
- 代码存在但调用链不完整，无法确认是否可达

标记为 ACTIVE（正常作为功能点）：
- 有明确的处理逻辑
- 调用链完整可追溯

**对 DEAD 条目的处理：**
不将其纳入功能点列表，但在笔记末尾的【跳过的内容（死代码）】章节中记录，
说明跳过原因，避免后续阅读者疑惑为什么没有对应功能点。

---

**Step 4：梳理模块结构与调用关系**

基于 Step 2 建立的完整文件清单，梳理：
- 文件目录结构（只列本次任务相关的部分）
- 主要模块划分（页面层 / 组件层 / 网络层 / 数据模型层）
- 模块之间的调用关系和数据流向
- 依赖的第三方库（若有）

---

**Step 5：拆解功能点**

基于 Step 2 的交互扫描结果（仅 ACTIVE + UNCERTAIN 条目）拆解功能点。

**拆解原则：**

- 若涉及做一个独立的页面：基础框架（页面骨架、路由注册、Tab 配置请求）作为一个功能点，让应用能够出现该空白页面
- 若涉及发请求获取数据：数据模型、解析、基础展示、组件基础布局作为一个功能点（禁止将"发请求"和"数据模型设计"拆成两个功能点）
- 若涉及交互功能相对独立的组件，可单独作为一个功能点
- 若涉及复杂的ui展示，"样式细节实现"作为独立功能点，作为最后一个功能点，用于记录ui细节：组件布局、样式、动画等
- 业务逻辑强耦合的可并为一个功能点
- 按开发依赖顺序排列，被依赖的排在前面


**数据链路完整性自查清单**（拆解完成后必须检查）：
对于涉及网络请求的功能点，逐一确认：
- [ ] 该功能点是否包含：请求发起 → 参数构建 → 返回解析 → 数据模型 → 基础UI展示 的完整链路？
- [ ] 若该链路被拆分到多个功能点，是否因为某个环节（如数据模型）被多处复用？
- [ ] 拆分后，每个功能点的开发笔记是否会出现"缺少上下文"的问题？

**严格检查（拆解完成后自查）：**
对照 Step 2 的完整交互元素列表，逐一确认每个 ACTIVE/UNCERTAIN 条目
都能在某个功能点中找到对应描述，没有遗漏。

---

**Step 6：写入笔记**

将分析结果写入 .ios-notes/task-overview.md，格式见【输出格式】章节。

## Supplement 模式：补充与修正任务总览

根据用户对功能点列表的反馈，修正 .ios-notes/task-overview.md。
可能的操作包括：补充漏掉的功能点、删除冗余功能点、
澄清 UNCERTAIN 状态、拆分或合并功能点。

### 输入

从调用方 prompt 中读取：
- mode: supplement
- user_feedback：用户的原始反馈内容

读取以下文件：
- .ios-notes/task-overview.md：当前功能点列表
- .migration/request.md：获取 iOS 项目路径（IOS_ROOT）

---

### 执行步骤

**Step 1：解析用户意图**

从 user_feedback 中提取所有修改意图，分类整理：

| 意图类型 | 示例 | 是否需要读 iOS 代码 |
|---|---|---|
| 补充功能点 | "漏了搜索功能" | 是 |
| 澄清 UNCERTAIN | "那个刷新是有用的" | 是 |
| 拆分功能点 | "F-002 太大，列表和筛选要分开" | 是 |
| 删除功能点 | "F-003 不需要迁移" | 否 |
| 合并功能点 | "F-003 和 F-004 是同一个东西" | 否 |
| 修改描述 | "F-001 的概述不准确" | 否 |

---

**Step 2：执行需要读 iOS 代码的操作**

对 Step 1 中需要读 iOS 代码的意图：

1. 从 user_feedback 提取关键词（功能名称、页面名、类名等）
2. 在 IOS_ROOT 中使用 Glob + Grep 定位相关文件
3. 快速阅读相关文件，明确：
   - 功能点的主要文件列表
   - 涵盖的交互入口
   - 与已有功能点的依赖关系
   - 状态判断（ACTIVE / UNCERTAIN）

注意：此处只做定位和范围确认，不做深度实现分析
（深度分析是 ios-feature-note 的工作）。

---

**Step 3：执行不需要读 iOS 代码的操作**

直接修改 task-overview.md：
- 删除：直接从功能点列表中移除对应条目
- 合并：保留较小 feature-id 的条目并更新描述，直接移除另一个条目
- 修改描述：直接更新对应字段

---

**Step 4：更新 task-overview.md**

补充功能点时，在功能点列表末尾追加新条目：
  feature-id 接续现有最大编号（如已有 F-003，新增为 F-004）
  格式与已有条目保持一致（见【输出格式】中的功能点条目格式）

澄清 UNCERTAIN 时：
  将对应条目的状态改为 ACTIVE 或 DEAD
  更新疑问字段，注明用户确认内容
  若改为 DEAD，移入【跳过的内容（死代码）】章节

拆分功能点时：
  将原条目替换为两个新条目
  保留原 feature-id 给第一个，新增编号给第二个
  在两个条目的依赖字段中标注关系
