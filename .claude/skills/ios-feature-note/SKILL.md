# Skill：iOS 功能点笔记

## 我的职责

深度分析 iOS 项目中某一个具体功能点的实现细节，
生成足够让鸿蒙开发者独立实现同等功能的参考笔记。

笔记必须以"让鸿蒙开发者无需再查看 iOS 源码，也能实现该功能点"为标准。

注意：
- 不要机械套用固定章节模板
- 要根据当前功能点的实际职责，自适应组织笔记结构
- 若当前功能点偏重网络请求，就重点写清请求链路
- 若当前功能点偏重数据模型，就重点写清字段结构、解析与映射关系
- 若当前功能点偏重 UI，就重点写清 UI 结构、状态变化和数据消费方式
- 若当前功能点同时涉及多个维度，则按实际重要性组织内容，确保信息完整

---

## 输入

从调用方 prompt 中读取：
- feature_id：当前功能点 ID（如 F-002）
- mode：initial（初次分析）或 supplement（补充分析）
- supplement_target：需要补充的具体内容（仅 supplement 模式时存在，
  来自进度日志中 `hm_dev[need_ios_info] 缺少：{xxx}` 的内容）

---

## 执行步骤

**Step 1：读取上下文**

读取以下文件：
- .ios-notes/task-overview.md：找到该功能点的主要文件列表
- .migration/request.md：获取 iOS 项目路径（IOS_ROOT）
- （supplement 模式）.ios-notes/note-{feature-id}.md：了解已有内容，明确缺失部分

---

**Step 2：阅读 iOS 代码**

initial 模式：全面阅读该功能点相关的所有文件
supplement 模式：聚焦 supplement_target 指定的内容，定向阅读相关文件

无论哪种模式，都需要先判断该功能点的主要职责属于哪一类：
- UI 展示类
- 网络请求类
- 数据模型 / 解析类
- 交互流程类
- 混合类（同时涉及多个方面）

然后按该功能点的实际职责决定笔记重点，但必须覆盖与实现直接相关的信息。

必须重点关注以下维度：

### 1. UI / 交互维度（若该功能点涉及页面或组件）
- 页面 / 组件的布局结构和层级
- 各 UI 元素的交互行为
- 状态切换时 UI 如何变化
- 依赖哪些数据字段驱动展示
- 是否存在条件渲染、动态样式、动态文案、禁用态、空态等

### 2. 数据请求维度（若该功能点涉及请求）
若当前功能点包含接口请求、请求封装、请求触发时机中的任一项，必须详细写清以下内容：

- 请求是从哪里发起的
- 请求由哪个类 / 方法封装
- 请求方式（GET / POST / 其他）
- 接口路径 / 协议号 / 路由标识（代码里能找到什么就写什么）
- 请求头 / 公共参数 / 签名参数（若有）
- 请求参数有哪些，分别从哪里来
- 参数是固定值、页面状态值、父组件传入值，还是其他接口返回值
- 请求触发时机（首屏加载、下拉刷新、切 tab、点击按钮、滚动分页等）
- 是否存在防抖、节流、去重、取消请求、串行 / 并行请求逻辑

禁止只写"调用 xxx 接口获取数据"这种粗略描述，必须写清调用链路。

### 3. 返回值解析维度（若该功能点涉及响应处理）
若当前功能点包含响应解析、字段转换、模型构建中的任一项，必须详细写清以下内容：

- 返回值原始结构大致是什么
- 成功 / 失败是通过哪个字段判断
- data / list / extra / pagination 等关键字段结构
- 哪些字段被使用，哪些字段被忽略
- 是否存在字段改名、默认值补齐、空值处理、类型转换
- 是否存在单位换算、枚举值映射、布尔值转换、时间格式处理
- 响应结果最终被转换成什么模型 / ViewModel / section 数据
- 转换逻辑在哪个文件、哪个方法里完成

如果代码中存在"响应结构 → 中间模型 → 展示模型"的多层转换，必须把整条链路串起来说明。

### 4. 数据传递维度（若该功能点涉及数据流转）
若当前功能点涉及数据在多层之间流转，必须说明：

- 数据从哪一层传到哪一层
- 是通过回调、属性、delegate、notification、binding、state、store 还是其他方式传递
- 谁持有数据
- 谁负责更新数据
- 谁消费数据并更新 UI
- 数据变化后如何触发界面刷新

### 5. 本地数据 / 缓存维度（若有）
- 是否使用内存缓存、磁盘缓存、UserDefaults、数据库等
- 缓存 key / 存储对象 / 读取时机 / 失效策略
- 本地缓存与网络请求的优先级关系

### 6. 业务逻辑维度
- 主流程步骤（用户操作 → 系统响应）
- 所有交互状态（初始 / 加载中 / 成功 / 失败 / 空数据）
- 失败处理和兜底逻辑
- 边界情况（空数据、无权限、网络异常、字段缺失、重复点击等）

若某个逻辑跨多个文件，必须串联说明完整链路，不能只描述单个文件。
若某个功能点本身只负责其中一部分（例如只负责模型转换，不负责请求发起），必须明确写清它在整条链路中的位置和上下游关系。

---

**Step 3：写入笔记**

initial 模式：
- 创建 .ios-notes/note-{feature-id}.md
- 根据当前功能点的实际职责，自行组织最合适的笔记结构
- 不要求固定章节顺序
- 但必须保证与该功能点实现直接相关的信息写完整、写具体、可落地

supplement 模式：
- 更新 .ios-notes/note-{feature-id}.md 中缺失的内容
- 优先补齐被明确追问的部分
- 不重复已有内容，不覆盖已有的准确描述

写作要求：
- 如果当前功能点偏"网络请求"，笔记重点应围绕请求入口、参数、响应、解析、状态处理展开
- 如果当前功能点偏"数据模型"，重点应围绕字段结构、解析逻辑、映射关系、上下游依赖展开
- 如果当前功能点偏"UI 展示"，重点应围绕布局结构、数据依赖、状态切换、交互逻辑展开
- 如果当前功能点是混合型，按实现重要性排序组织内容，但不能遗漏关键链路

---

## 输出要求

写入路径：.ios-notes/note-{feature-id}.md

笔记结构不要求固定模板，允许根据功能点职责自由组织。
但必须满足以下要求：

### 必须包含的基础信息
- 功能点名称和 feature-id
- 相关源文件列表
- 当前功能点在整体链路中的职责
- 与其他功能点 / 文件的上下游关系

### 若涉及 UI，必须写清
- 界面结构
- 关键组件
- 交互行为
- 状态变化
- 数据如何驱动 UI

### 若涉及网络请求，必须详细写清
- 请求发起位置
- 请求调用链路
- 请求方式
- 接口路径 / 协议号 / 标识信息
- 请求参数、参数来源、参数含义
- 请求触发时机
- 成功 / 失败判断方式
- 错误处理逻辑

### 若涉及返回值处理，必须详细写清
- 响应结构
- 关键字段
- 字段含义
- 字段转换 / 默认值 / 类型处理
- 响应如何被解析成模型或展示数据

### 若涉及数据模型，必须详细写清
- 模型名称
- 字段列表
- 字段类型
- 字段来源
- 模型之间的映射关系
- 模型如何传递给后续层使用

### 若涉及数据流转，必须写清链路
- 数据从哪里来
- 在哪里解析
- 在哪里存储
- 传递给谁
- 谁最终消费
- 如何触发 UI 更新

### 若涉及状态与边界处理，必须写清
- 初始态
- 加载态
- 成功态
- 失败态
- 空态
- 特殊异常场景

### 写作质量要求
- 不允许只写结论，必须写实现链路
- 不允许只写接口名，必须写参数和返回值处理
- 不允许只写模型名，必须写字段结构和映射关系
- 不允许只写"数据传给页面"，必须写清通过什么机制传递
- 不粘贴完整源码，用文字说明逻辑
- 若一个逻辑跨多个文件，必须串联说明完整链路

### 参考章节结构（仅供参考，非强制）

以下结构可作为参考，但可根据实际情况调整、合并、新增或删减章节：

- 功能点名称与职责
- 相关源文件
- UI 结构（若有）
- 数据来源与流转（若有）
- 网络请求详情（若有）
- 返回值解析与映射（若有）
- 数据模型（若有）
- 主流程
- 交互状态
- 边界情况
- 鸿蒙实现注意事项

## 约束

- 只读 IOS_ROOT 中的文件，禁止修改任何 iOS 源码
- 只写 TOOL_ROOT/.ios-notes/ 目录下的文件
- 不粘贴完整源码，用文字描述逻辑
- 不要为了套模板而强行组织章节，笔记结构应服务于当前功能点的实际职责
- 只要功能点涉及数据请求、数据解析、数据模型、数据流转中的任一项，就必须详细写清
- "详细写清"指的是：让鸿蒙开发者不需要再回看 iOS 源码，也能完成同等实现
- 禁止使用"请求接口获取数据""解析后展示到页面"这类过于笼统的描述
- 对数据相关逻辑，必须尽可能写到：请求入口、参数来源、返回结构、解析过程、模型映射、传递路径、消费位置
- 若某个功能点只负责整体链路的一部分，必须明确写清它在链路中的位置和与其他功能点的协作关系
- 跨多个文件的逻辑必须串联说明完整链路，不能只描述单个文件的行为
- 笔记的衡量标准：鸿蒙开发者不需要再看 iOS 源码就能实现同等功能
